# 10. 비동기 이벤트 기반 Saga 패턴 구현 및 트러블슈팅

본 자료는 병렬적 코레오그래피 Saga 패턴을 실제 구현하면서 마주한 기술적 난제들과 이를 해결한 실전 전략을 정리합니다.

## 1. 구현 목표: 병렬적 코레오그래피 (Parallel Choreography)

기존의 순차적 방식에서 벗어나, 주문 생성 시 재고 차감과 적립금 차감을 **동시에(Parallel)** 진행하여 시스템의 응답 속도를 극대화하고 서비스 간 결합도를 낮추는 것을 목표로 합니다.

## 2. 실전 트러블슈팅 (Troubleshooting)

### 2-1. Jackson 역직렬화 에러 (Deserialization Issue)
*   **문제**: Kafka를 통해 메시지를 수신할 때 `OrderCreatedEvent` 객체를 생성하지 못하는 에러 발생.
*   **원인**: Jackson 라이브러리는 JSON을 객체로 변환할 때 **기본 생성자(Default Constructor)**를 필요로 하지만, Lombok의 `@RequiredArgsConstructor`만 사용하여 기본 생성자가 누락됨.
*   **해결**: 모든 이벤트 클래스에 `@NoArgsConstructor(access = AccessLevel.PROTECTED)`를 추가하여 불변성을 유지하면서도 역직렬화가 가능하도록 수정.

### 2-2. 병렬 업데이트 동시성 문제 (Lost Update)
*   **문제**: 재고 성공과 포인트 성공 메시지가 거의 동시에 도착할 때, 한쪽의 업데이트가 다른 쪽의 업데이트를 덮어쓰는 현상 발생 (데이터 정합성 깨짐).
*   **원인**: 비동기 환경에서 주문 엔티티의 상태를 변경할 때, 동일한 시점에 DB에서 데이터를 읽어와 수정 후 저장하면서 발생하는 전형적인 동시성 이슈.
*   **해결**: 
    1.  **낙관적 락(Optimistic Locking)**: 주문 엔티티에 `@Version` 필드를 추가하여 충돌 감지.
    2.  **재시도 로직(Retry)**: 충돌 발생 시(`OptimisticLockingFailureException`) 즉시 포기하지 않고 3회까지 재시도하여 최종적으로 두 성공 소식이 모두 반영되도록 보장.

### 2-3. DB 예약어 및 테이블 관리
*   **문제**: `orders` 테이블 명이 SQL 예약어와 충돌하여 쿼리 실행 시 예외 발생 가능성.
*   **해결**: 테이블 명을 `eda_order`로 변경하고, 기존의 낡은 테이블들을 정리하여 깨끗한 DB 스키마 상태 유지.

## 3. 최종 아키텍처 구조 분석

### 🧱 핵심 컴포넌트
- **eda-common**: 서비스 간 약속된 **이벤트(Event)**와 공통 예외 정의. 모든 서비스의 통신 규약 역할.
- **eda-order-service**: Saga의 **오케스트레이터(State Holder)** 역할. 주문의 최종 상태와 각 단계별 성공 여부 관리.
- **eda-stock/point-service**: 독립적인 **워커(Worker)** 역할. 자신의 비즈니스 로직만 수행하고 결과만 Kafka로 전송.

### 🔄 메시징 흐름 (Success Scenario)
1.  **Order**: `OrderCreatedEvent` 발행 -> DB에 `PENDING` 저장.
2.  **Stock & Point**: (병렬) 이벤트 수신 -> 각각 차감 로직 수행 -> `DeductedEvent` 발행.
3.  **Order**: (병렬) 결과 수신 -> **낙관적 락과 재시도**를 거쳐 `isStockDeducted`, `isPointDeducted` 업데이트 -> 둘 다 완료 시 `COMPLETED` 전환.

## 4. 시사점: 비동기 시스템의 정석

비동기 시스템은 단순히 메시지를 주고받는 것보다 **"실패했을 때 어떻게 복구할 것인가"**와 **"동시에 들어오는 요청을 어떻게 안전하게 처리할 것인가"**가 훨씬 중요합니다. 이번 구현을 통해 우리는 성능(병렬성)과 정합성(낙관적 락)이라는 두 마리 토끼를 모두 잡는 설계를 완성했습니다.
