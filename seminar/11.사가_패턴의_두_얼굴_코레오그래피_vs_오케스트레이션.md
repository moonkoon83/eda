# 11. 사가 패턴의 두 얼굴: 코레오그래피 vs 오케스트레이션

본 자료는 마이크로서비스 아키텍처(MSA)에서 분산 트랜잭션을 관리하는 **사가(Saga) 패턴**의 두 가지 핵심 구현 방식과, 우리 프로젝트가 채택한 **이벤트 기반 병렬 코레오그래피**의 가치를 분석합니다.

---

## 1. 사가 패턴(Saga Pattern)이란?

분산 환경에서 여러 서비스에 걸친 트랜잭션을 관리하기 위해, 각 서비스의 로컬 트랜잭션을 순차적으로 실행하고 실패 시 **보상 트랜잭션(Compensating Transaction)**을 통해 데이터 정합성을 맞추는 패턴입니다.

## 2. 코레오그래피(Choreography) vs 오케스트레이션(Orchestration)

| 비교 항목 | 코레오그래피 (Choreography) | 오케스트레이션 (Orchestration) |
| :--- | :--- | :--- |
| **제어 방식** | **분산 제어** (각자 판단) | **중앙 제어** (지휘자 존재) |
| **메시지 성격** | **Event** ("일어난 사실" 공유) | **Command** ("수행할 명령" 전달) |
| **결합도** | **매우 낮음** (서비스 간 무관심) | **중간** (지휘자가 참여자를 앎) |
| **비유** | 무용수들이 음악에 맞춰 춤추는 **안무** | 지휘자의 손짓에 따르는 **오케스트라** |

---

## 3. 왜 "이벤트 기반(Event-Driven)"인가?

현재 우리 구조는 `OrderCreatedEvent`라는 **과거형의 사실(Fact)**을 발행합니다.
- **장점**: 발행자(Order)는 수신자(Stock, Point)가 누구인지, 그들이 무엇을 하는지 알 필요가 없습니다. 
- **유연성**: 나중에 "알림 서비스"나 "로그 분석 서비스"가 추가되어도 기존 `OrderService` 코드는 단 한 줄도 수정할 필요가 없습니다. (Open-Closed Principle 준수)

## 4. 왜 "병렬(Parallel)" 구조인가?

순차적 사가(Sequential Saga)는 앞 단계가 끝나야 다음 단계로 넘어가지만, 병렬 구조는 **동시에** 작업을 시작합니다.
- **성능 최적화**: 전체 응답 시간 = `Max(재고 차감 시간, 포인트 차감 시간)`.
- **복잡도 대응**: 동시 업데이트로 인한 데이터 덮어쓰기 문제를 **낙관적 락(@Version)**과 **재시도(Retry)** 전략으로 극복했습니다.

## 5. 실전 아키텍처 분석 (우리 프로젝트의 모습)

### 🧱 핵심 매커니즘
1.  **발행 (Publishing)**: `OrderService`가 Kafka 토픽(`order.created`)에 이벤트를 던짐.
2.  **자율적 수행 (Autonomous Execution)**: `Stock`과 `Point` 서비스가 동시에 이벤트를 가로채 비즈니스 로직 수행.
3.  **결과 취합 (Join/State Management)**: `OrderService`가 각 서비스의 성공 소식을 기다렸다가 최종 상태를 `COMPLETED`로 확정.

---

## 6. 결론: 언제 어떤 사가를 선택해야 하는가?

- **코레오그래피(현재 방식)**: 서비스 간의 독립성이 중요하고, 전체 흐름이 아주 복잡하지 않을 때 최적의 선택입니다. 성능상 이점이 매우 큽니다.
- **오케스트레이션**: 비즈니스 로직이 거대하고 복잡하여 중앙에서 전체 워크플로우를 가시적으로 관리해야 할 때(예: 배달 앱의 복잡한 주문 흐름) 고려합니다.

> **현재 우리 프로젝트는 "이벤트 기반 병렬 코레오그래피"를 통해 현대적인 비동기 시스템의 성능과 확장성을 모두 확보했습니다.**
